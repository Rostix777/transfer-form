02 · CPS · Skill

CPS: Agent-Led Orchestration

Единый AI-guided процесс создания клиентских поручений через чат (Mobile / Web). Логика разносится по слоям — клиенты только рендерят, агент оркестрирует, бэкенд валидирует и подаёт.

Архитектурная модель · 3 слоя
Mobile / Web
UI-LAYER
  • • рендер wizard по Step Plan
  • • сбор ответов пользователя
  • • прогресс k/N, статус
  • • отправка wizard_result
  • — не формирует payload
  • — не вызывает submitCps
Agent (AI)
ORCHESTRATOR
  • • определяет cps_number
  • • владелец Step Plan
  • • prefill + маппинг ответов
  • • формирует payload
  • • ValidateAndSubmitDraft
  • — schema = источник истины
Backend
DETERMINISTIC
  • • GetFormSchemaByCpsNumber
  • • ValidateAndSubmitDraft
  • • валидация payload
  • • нормализация типов
  • • вызов submitCps(0)
  • — только тут вызов submitCps
Step Plan · Agent → Client
{
  "wizard_id": "w_8a3…",
  "cps_number": "CPS200",
  "schema_version": 3,
  "progress_total": 4,
  "steps": [
    { "field_key": "amount",
      "ui_component": "moneyInput",
      "required": true,
      "prefill": null }
  ],
  "signing_config": { … }
}
UI Components · MVP
textInput numberInput moneyInput checkbox select accountPicker summary · fallback → web-view
Главное: исключение дублирования логики на клиентах, безопасность через детерминированный backend, idempotency_key защищает от дублей.
03 · CPS · Security

Безопасность и Security Session

Чёткое разделение «подготовки черновика» и «юридического подписания». Draft создаётся без SID клиента, подпись — только в рамках security session.

01
Создание Draft (status=0)
Backend-слой ValidateAndSubmitDraft вызывает submitCps(status=0) — без SID клиента. Draft создаётся в неисполняемом статусе и не может быть проведён без подписи.
original_client_user_id wizard_id idempotency_key
02
Подписание · web-view review
Клиент открывает web-view review → инициализация Security Session → получение SID клиента → подпись и перевод status=0 → next. Без валидного SID нельзя подписать draft.
SID клиента не SID агента проверка прав подписи

05 · Voice AI

Голосовой ИИ и Звонки

Стандартное подключение через WebRTC — как у GPT и Gemini. Мгновенная реакция, естественный диалог, низкая задержка.

WebRTC peer connection
P2P audio пиринг, минимальная задержка, эхо-подавление, авто-уровень.
Естественный диалог
Прерывания, паузы, бэкчэннелы — пользователь говорит как с человеком.
Мгновенная реакция
Стриминговый STT/TTS, реакция на речь до завершения фразы.
Контур безопасности
Голос — только вход в чат-агента. Чувствительные действия → CPS + security session.
// pseudocode
const pc = new RTCPeerConnection(cfg);
pc.addTrack(localMicTrack);
const offer = await pc.createOffer();
await pc.setLocalDescription(offer);
// → POST /voice/sdp → returns answer
await pc.setRemoteDescription(answer);
06 · MCP

Интеграция через MCP

Три контура Model Context Protocol — для клиентов, для сторонних CRM и внутренний расширяемый.

5.1
Public MCP
для клиентов
Публичные API для клиентских ботов. Клиент строит свою автоматизацию поверх стабильного контракта.
  • • rate-limit per token
  • • OAuth scopes
  • • semver контракты
5.2
CRM Bridge MCP
сторонние CRM
Передача и сбор чатов / сообщений между системами. Один протокол — разные интеграции (Bitrix, amoCRM, Salesforce).
  • • webhook in / webhook out
  • • унифицированная схема message
  • • threading по client_id
5.3
Internal MCP
для ядра ИИ
Расширяемый протокол для работы с ядром AI. Постоянно растёт — новые скиллы подключаются без перерелиза клиентов.
  • • tool registry
  • • capability discovery
  • • zero-deploy расширение
